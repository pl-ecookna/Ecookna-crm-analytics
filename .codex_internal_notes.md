# Codex Internal Notes

Дата актуализации: 2026-05-29

## Текущая архитектура

- Монорепозиторий: `apps/api`, `apps/web`, `packages/shared-types`, `packages/api-client`.
- Основной поток: CRM webhook -> S3 -> Postgres -> Speech provider -> OpenAI -> Postgres -> Web UI.
- Провайдер речи выбирается на уровне окружения, а UI показывает метрики по сохраненному `provider` записи.
- Поток отказов: CRM webhook -> S3 -> Postgres -> Deepgram -> OpenAI -> Postgres -> Web UI.

## Активные сервисы

- API: Express 5, Node.js ESM.
- Web: React/Vite.
- DB: PostgreSQL через `pg`.
- Storage: S3-compatible storage.
- Speech: Yandex SpeechSense или SaluteSpeech для основного потока, Deepgram для отказов.

## Env Variables

- SpeechSense/SaluteSpeech: `SPEECH_PROVIDER`, `YANDEX_SPEECHSENSE_API_KEY` или `YANDEX_SPEECHSENSE_IAM_TOKEN`, `YANDEX_ORGANIZATION_ID`, `YANDEX_SPEECHSENSE_SPACE_ID`, `YANDEX_SPEECHSENSE_CONNECTION_ID`, `YANDEX_SPEECHSENSE_PROJECT_ID`, `SBER_AUTH_KEY`, `SBER_SCOPE`.
- Debug: `YANDEX_DEBUG_CALL_ID`, `YANDEX_DEBUG_POLL_TIMEOUT_MS`, `YANDEX_DEBUG_POLL_INTERVAL_MS`.

## Структура БД

- Основные таблицы: `crm_analytics`, `disaproov_calls`, `prompts`, `app_users`, `auth_sessions`.
- `transkription_full_json` хранит `provider`, raw-ответ и нормализованный `call_features`; БД изменений не требует.

## Deployment Notes

- Базовый deployment: Docker Compose/Dokploy.
- Для диагностики SpeechSense/SaluteSpeech можно запускать локально из `apps/api` без старта API-сервера.

## Технический долг

- `apps/api/src/config/env.js` валидирует env всего сервиса даже для узких debug-скриптов; локальный SpeechSense debug-скрипт обходит нерелевантные обязательные env через dummy defaults.

## Известные проблемы

- Пользователь сообщил: файл отправляется в SpeechSense, но результат не возвращается и визуально новый файл не виден в консоли Yandex.
- Локальная проверка с тестовыми env подтвердила: сразу после upload `gRPC TalkService.Upload` возвращает `talk_id`, но `/talks/get` по этому id может возвращать `talk: []`, а `/talks/search` не показывает новый id до завершения индексации.
- REST `/talks/search` и `/talks/get` для уже существующих диалогов работают с теми же `organization/space/connection/project`, значит базовые API key и scope валидны.
- Ответ поддержки Yandex: прямой upload не является поддерживаемым production-путем для появления файла в проекте SpeechSense.
- SpeechKit STT не подходит для целевого решения; целевой вариант - Object Storage ingestion/интеграция SpeechSense.
- Подтверждено на практике: итоговый `ID диалога` в UI/search может отличаться от `talk_id`, который возвращает `TalkService.Upload`.
- Подтверждено на практике: задержка до появления диалога в project UI/search может составлять около 1 часа. Для `speechsense-debug-20260528080414` оценка составила примерно 1 час 3 минуты.

## Последние решения

- Добавлен `scripts/debug-yandex-local-file.js`: минимальный пайплайн `local file -> SpeechSense Upload -> talks/get polling -> talks/search visibility diagnostics`.
- Добавлен gRPC upload deadline `YANDEX_UPLOAD_TIMEOUT_MS`, чтобы прямой upload не зависал без диагностики.
- В UI и типах добавлен provider-aware рендер: метрики показываются по провайдеру, сохраненному в `transkription_full_json.provider`.

## Текущий статус реализации

- Диагностический скрипт добавлен и прогнан с тестовыми Yandex env.
- Визуализация речевых метрик теперь адаптируется под Sber/Yandex без отдельного блока дополнительных метрик.
- Перед анализом следующего speech-provider зафиксированы факты по текущему поведению SpeechSense upload/indexing.
